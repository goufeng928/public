本文将通过 XX市XX区人民医院HRP信息系统建设项目实例 来论述 `范围管理` 在其中的作用。

## 范围管理定义

**信息系统项目管理师教材定义**:

```txt
项目范围管理是为了达到项目目标, 交付具有某种特质的产品和服务, 项目所规定要做的工作。
项目范围管理就是要确定哪些工作是应该做的, 哪些不应该包括在项目中。
项目范围是项目目标的更具体的表达。
```

**壹句话总结**:

```txt
项目范围管理是为了达到项目目标, 交付具有某种特质的产品和服务, 项目所规定要做的工作。
简单地理解项目范围管理就是要做范围内的事, 而且只做范围内的事, 既不少做也不多做。
```

## 如何进行项目范围管理

**(1) 规划范围管理**:

```txt
规划范围管理:
规划范围管理是编制范围管理计划, 书面描述将如何定义、确认和控制项目范围的过程, 其主要作用是在整个项目中对如何管理范围提供指南和方向。

规划范围管理的技术:
专家判断、会议等

规划范围管理的输出:
范围管理计划、需求管理计划
```

**(2) 收集需求**:

```txt
收集需求:
收集需求是为实现项目目标而确定、记录并管理干系人的需要和需求的过程, 其作用是为定义和管理项目范围 (包括产品范围) 奠定基础。

收集需求的方法:
访谈、焦点小组、原型法、系统交互图等

收集需求的输出:
需求文件、需求跟踪矩阵
```

**(3) 定义范围**:

```txt
定义范围:
定义范围是制定项目和产品详细描述的过程, 其主要作用是明确所收集的需求哪些将包含在项目范围内, 哪些将排除在项目范围外, 从而明确产品、服务或成果边界。

定义范围的技术:
产品分析、备选方案生成、引导式研讨会等

定义范围的输出:
项目范围说明说、项目文件更新
```

**(4) 创建 WBS**:

```txt
创建 WBS:
创建 WBS 是将项目可交付成果和项目工作分解成较小的、更易于管理的组件的过程, 其主要作用是对所要交付的内容提供一个结构化的视图。

创建 WBS 的技术:
分解、专家判断等

创建 WBS 的输出:
范围基准、项目文件更新
```

**(5) 确认范围**:

```txt
确认范围:
确认范围是正式验收已完成的可交付成果的过程, 其主要作用是使验收过程具有客观性, 同时, 通过验收每个可交付成果, 提高最终产品、服务或成果获得验收的可能性。

确认范围的技术:
检查、群体决策技术

确认范围的输出:
验收的可交付成果、变更请求、工作绩效信息
```

**(6) 控制范围**:

```txt
控制范围:
控制范围是监督项目和产品的范围状态、管理范围基准变更的过程, 其主要作用是在整个项目期间保持对范围基准的维护。

控制范围的技术:
偏差分析

控制范围的输出:
工作绩效信息、变更请求
```

## XX市XX区人民医院HRP信息系统建设项目(范围管理)实例

**项目背景**:

```txt
# 2023 年 9 月, 我公司通过公开招标的方式中标了 XX 市 XX 区人民医院 HRP 信息系统建设项目, 我作为乙方项目经理全程参与了该项目的建设工作。
  本项目合同金额 3,630,000.00 元, 主要建设内容包括基础平台系统、财务会计系统、全面预算管理系统、科室成本核算系统、项目成本核算系统、病种成本核算系统、固定资产管理系统、合同管理系统、人力资源管理系统、统一对账系统、智能发放系统、报销管理系统、移动应用、决策分析(BI)、接口服务的建设。
  建设工期 8 个月, 前 5 个月内完成项目实施建设, 后 2 个月上线试运行, 最终验收 1 个月。
  该项目采用 B/S 架构, 前端展现包括系统入口、总览和数据仪表盘、小程序和移动APP, 后端主要采用Java、C#等语言开发模块, 实现预算编制模块、成本控制模块、资源管理模块、财务分析模块、凭证管理模块、成本核算模块、员工信息模块、薪酬绩效模块、报表与分析模块、集成模块等。
  关键技术指标要求实现系统的用户、权限、引入、输出、日志等功能, 要求支持 EAI 集成, 要求实现 PDM 系统接口、配置、规则等, 要求实现财务会计和预算会计双重功能平行记账、支持新会计制度要求的相关会计报表的生成, 要求实现资产条码管理, 报销单自动分摊、归集等。
  该项目于 2024 年 5 月顺利完工并通过客户验收, 获得了客户的肯定。
  本文以该项目为例, 结合作者实践经验, 探讨信息化系统项目建设过程中在范围管理方面经常遇到的问题及解决方法。
  主要就如何做好规划范围管理, 收集需求, 定义范围, 创建 WBS, 确认范围, 控制范围这六个方面来进行阐述。最后总结项目中的成功经验, 以及存在的不足之处和具体改进措施。
```

**范围管理论述**:

```txt
# 鉴于该项目对公司意义重大, 公司为此组建了单独的项目团队并任命我为项目经理。
  收到委任后, 我仔细分析了项目合同、工作说明书后得出: 本项目范围广、工期紧、干系人众多, 需求不明。
  怎样规划范围管理为范围管理工作提供指南和方针; 怎样收集需求和定义范围以明确项目边界; 怎样控制范围以防止范围蔓延, 这些都是本项目需要重点考虑的问题。
  因此我认为有效的范围管理是该项目成功的关键。

# 由于范围管理在该项目中显得尤为重要, 作为项目经理我在本项目中除了对其它各管理领域恪尽职守外, 还特别对范围管理从以下几个方面进行了管理:

# 一、规划范围管理
* 规划范围管理就是根据项目实际情况编制一份有效的范围管理计划, 为整个项目中如何管理范围提供指南和方针。
* 在这一过程中, 我带领团队成员对项目合同和工作说明书进行了深入理解。
  通过开会讨论的方式确定出怎样收集、定义、分解、确认和控制项目范围的方法。
  最后结合项目管理计划和项目章程等因素, 根据公司现有模板制订出了范围管理计划。

# 二、收集需求
* 收集需求就是为实现项目目标而确定、记录并管理干系人需求的过程。事实证明大多数项目失败的原因都是因为需求不明确所导致。
* 因为行业壁垒的原因, 大部分干系人都无法完整地描述自身需求, 因此只有与干系人有效合作才能成功收集需求。
  在这一过程中, 我带领需求分析师雷工与客户信息科主任李工进行了深入交流, 让我们对项目主要应用场景、工作内容、业务流程等均作了详细了解。
  我们还一起深入到客户各个科室中, 通过采用面对面交谈、问卷调查等方法详细了解了用户的需求。
  我们采用需求跟踪矩阵来对客户的需求进行记录和跟踪, 下面以表格的形式简要列出项目核心需求跟踪矩阵: 
  +-------------+---------+---------------------------+
  |用户需求     |变更状态 |工作成果                   |
  +-------------+---------+---------------------------+
  |系统入口     |原始     |Web 站点                   |
  |移动端展现   |原始     |微信小程序, 移动 APP       |
  |EAI 集成     |原始     |模块接口、EAI 接口说明文档 |
  |PDM 系统接口 |原始     |模块接口、PDM 接口说明文档 |
  |财务分析     |原始     |服务端相应功能模块         |
  |数据分析     |原始     |服务端相应功能模块         |
  |资产条码管理 |原始     |服务端相应功能模块         |
  |员工管理     |原始     |服务端相应功能模块         |
  +-------------+---------+---------------------------+

# 三、定义范围
* 收集完需求后, 我们结合项目合同、范围管理计划以及收集到的需求等, 综合分析后制定了项目范围说明书并得到客户确认。

# 四、创建 WBS
* 创建 WBS 就是将可交付成果和项目工作分解为较小的更易于管理的组件, 其主要作用就是对所要交付的内容提供一个结构化视图。
* 在本项目分解过程中我们借助 Microsoft Project 项目管理软件, 自上而下逐层分解, 遵循 8/80 小时原则, 一般分解控制在 4 - 6 层。
  下面简要描述本项目 WBS: 
  +----------+----------------------+---------+-------+
  |工作编号  |工作任务              |工期(天) |责任人 |
  +----------+----------------------+---------+-------+
  |0         |HRP 信息系统建设项目  |240      |苟风   |
  |1         |项目启动              |6        |苟风   |
  |2         |需求调研与分析        |15       |雷晗   |
  |3         |系统设计、搭建、开发  |114      |何林   |
  |3.1       |系统设计              |24       |何林   |
  |3.1.1     |概要设计              |10       |何林   |
  |3.1.1.1   |过程设计              |2        |何林   |
  |3.1.1.1.1 |程序流图设计          |1        |何林   |
  |3.1.1.1.2 |判定表                |1        |何林   |
  |3.1.2     |详细设计              |14       |何林   |

  |5         |系统试运行            |45       |王薪   |
  |6         |项目终验              |30       |苟风   |
  +----------+----------------------+---------+-------+

# 五、确认范围
* 确认范围应该贯串项目始终, 在项目的每个阶段末, 我都会对阶段可交付成果先进行检查。检查其是否可确认, 是否满足产品要求、是否符合质量标准等等。
  一切准备就绪后, 我会组织包括甲方人员在内的干系人, 对照项目合同和范围说明书等, 对可交付成果进行查验和确认。

# 六、控制范围
* 控制范围就是监督项目和产品的范围状态, 管理范围基准变更的过程, 其主要作用就是在整个项目期间保持对范围基准的维护。
* 在这一过程中, 我会通过收集各项数据以判断项目范围是否已发生变化。
  我还会对变更请求先进行预审, 确定变更的必要性和可行性后, 再提交 CCB (变更控制委员会) 批准后才进行实施。
  如果范围基准发生了变化, 我还要对项目范围基准进行更新。
  我们还对所有变更进行记录和归档, 以备将来使用。

# 经过我们团队历时 8 个月的辛苦努力, 本项目于 2024 年 5 月如期顺利完工并通过客户验收。
  总结整个项目的实施, 我认为本项目的成功得益于我范围管理的成功运用。
  通过科学的规划并编制了一份切实可行的范围管理计划, 使得范围管理工作能有条不紊的进行; 通过定义和分解范围, 使得项目范围更加清晰明确; 通过持续的监控项目范围状态, 使得范围基准始终受到控制。
  通过这一系列方法的实施, 建立了一套高效的范围管理机制, 为项目目标的成功实现提供了有效保障。
* 当然还是有不足之处, 比如 Web 前端开发人员未经许可, 私自增加了图表主题换肤功能。我对其进行了思想教育, 并要求各小组长今后要严格审查代码, 谨防问题再次发生。
  在今后的工作中我将更加注重这方面的管理, 不断的学习充电, 努力提升自身管理水平, 为我国的信息化建设贡献自己的力量。
```
